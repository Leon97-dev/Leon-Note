# 현업에서 자주 쓰는 폴더명

# core/ (가장 실무스럽고 깔끔함)

"애플리케이션의 핵심 기능들"이라는 의미.

```bash
src/core/error/
src/core/logger/
src/core/env/
```

# common/

“공통 기능 모음”이라는 의미.
→ 많은 스타트업이 선호
→ 간단한 프로젝트에 딱 맞음

# shared/

“여러 계층에서 공유되는 코드들”
→ TypeScript, 모노레포, 대형 규모에서 자주 등장

# utils/

전통적으로 많이 사용되는 이름.
단점
너무 가볍고 의미가 넓음
logger/error 같은 “코어 기능”을 넣기엔 약함
그래도 작은 프로젝트에서는 자주 씀.

# lib/

라이브러리처럼 재사용되는 코드.
보통 DB client, API client 등이 들어감.

# infra/

“외부와 연결된 모든 인프라 계층”

- logger
- db
- prisma/orm
- redis
- smtp
- cloud storage
- external API client

# modules

각 도메인을 독립적인 모듈로 관리하는 방식
큰 회사에서 자주 봄

```bash
src/modules/user/
src/modules/auth/
src/modules/payment/
```

NestJS 프로젝트는 아예 modules 구조가 기본임.

# domain

DDD(Domain Driven Design) 채택한 회사에서 등장
도메인 중심 설계.

```bash
src/domain/user/
src/domain/order/
src/domain/product/
```

# application

서비스/유즈케이스 계층을 담당
DDD 혹은 Clean Architecture 스타일.

# adapters

외부 시스템과 통신하는 계층
(controllers, event handlers, http adapters 등)

# services vs usecases

조금 더 깔끔하게 나누고 싶을 때 usecases라는 이름 사용.

# config

Express/DB/CORS/Helmet 등 설정 전용
이건 Express 프로젝트에서 거의 100% 있음.

# src

그냥 메인 폴더임 무조건!
