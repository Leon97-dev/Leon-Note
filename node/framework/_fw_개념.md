# 프레임워크(Framework) 개념

1️⃣ 정의

프로그램의 기본 구조와 실행 흐름을 미리 설계해둔 뼈대
개발자가 모든 걸 처음부터 짜지 않아도,
정해진 규칙과 흐름 안에서 코드를 작성하면 전체 시스템이 자동으로 돌아가게 만들어주는 개발 틀이다.

2️⃣ 특징

**< 내 코드가 프레임워크 안에서 실행된다 >**
라이브러리는 내가 직접 호출(call) 하는 도구
프레임워크는 나를 호출한다 (Inversion of Control)
→ 이걸 “IoC” 또는 “제어의 역전”이라고 부른다.

```js
// 라이브러리 예시 (내가 호출)
import dayjs from 'dayjs';
const now = dayjs().format();

// 프레임워크 예시 (프레임워크가 호출)
app.get('/users', controller.getUsers);
```

Express가 라우트를 등록하고, 요청이 오면 내 함수를 불러주는 구조
→ 즉, 내가 주도하는 게 아니라 프레임워크가 나를 부른다.

**< 코드 구조가 강제된다 >**
파일 구조, 함수명, 라우팅, 라이프사이클 등 프레임워크가 미리 정의해둔 규칙을 따라야 한다.
대신 일관된 설계, 유지보수 용이성, 생산성 향상이 생긴다.

**< 확장보다 설정이 중요 >**
프레임워크는 “틀”을 제공하므로 개발자는 틀 안에서 설정(Configuration)을 통해 동작을 조정한다.
예: NestJS의 데코레이터(@Controller, @Get) → “무엇을 어떻게 동작시킬지”를 선언식으로 설정

3️⃣ 대표적인 프레임워크 예시

- `백엔드`: Express, NestJS, Fastify
  \_ Node.js 서버 프레임워크

- `프론트엔드`: Next.js, Nuxt.js, Remix
  \_ React/Vue 기반 SSR(서버사이드렌더링) 프레임워크

- `테스트`: Jest, Mocha
  \_ 테스트 실행과 라이프사이클 제어

- `전체 애플리케이션 구조`: Electron, Expo
  \_ 데스크톱·모바일 앱 전체 틀 제공

4️⃣ 핵심 철학 — IoC & Convention over Configuration

**IoC (Inversion of Control)**
제어의 역전: 개발자가 프레임워크 내부 동작을 직접 제어하지 않고, 프레임워크가 개발자의 코드를 적절히 호출한다.

**Convention over Configuration**
설정보다 규칙(convention) 을 우선시한다.
규칙을 따르면 설정이 줄고, 생산성이 높아진다.

6️⃣ 장점과 단점

✅ 장점: 일관된 코드 구조, 빠른 개발 속도, 커뮤니티 & 문서 풍부, 유지보수 용이

❌ 단점: 자유도가 낮음, 학습 곡선이 있음, 프레임워크 버전에 종속, 커스터마이징 어려움
